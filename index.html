Section 0 · Why DynamoDB exists

The story: Amazon's shopping cart broke

In 2004, Amazon ran its shopping cart on Oracle (a relational database). During the holiday shopping rush, the database could not keep up. The site went down. Amazon lost money every minute.

Amazon's engineers realized something important: the shopping cart does not need SQL. It only needs two operations:

Put

"Save this cart for user 42."

Get

"Give me the cart of user 42."

No JOINs. No complex queries. Just key → value. So they built a simple key-value store that could scale by adding more machines. That system became the Dynamo paper (2007). Later, AWS turned it into a managed service: DynamoDB (2012).

Real system · Amazon.com itself

Today Amazon runs its shopping cart, order history, and huge parts of the retail site on DynamoDB. In 2019 Amazon shut down its last internal Oracle databases.

Prime Day 2021 numbers (from AWS's own paper): DynamoDB served trillions of API calls and peaked at 89.2 million requests per second — with single-digit millisecond latency. No human touched a server. That is the scale this guide explains.

Section 1 · Basics

What exactly is DynamoDB?

DynamoDB is a database from AWS. Three words describe it:

NoSQL (key-value)

You find data by its key. Like a giant HashMap that lives across thousands of machines. No JOINs, no SQL.

Fully managed

You never see a server. You just call an API. AWS handles machines, disks, replication, and failures.

Fast at any size

A read takes a few milliseconds. Whether your table has 1 GB or 100 TB — same speed. This is the main promise.

Not for everything

Bad for ad-hoc reporting, complex JOINs, "search by anything". You must know your queries before you design the table.

Real system · Zoom during COVID

In early 2020, Zoom's daily meeting participants jumped from 10 million to 300 million in about three months. A 30x jump. No team can buy and set up servers that fast.

Zoom used DynamoDB for user and meeting metadata. Because DynamoDB adds capacity by itself, Zoom scaled 30x without redesigning the database. That is what "fully managed" buys you: the scaling problem becomes AWS's problem.

Section 2 · Keys

Partition key and sort key — in plain words

Think of a huge library with thousands of shelves.

Partition key = which shelf

DynamoDB hashes this value to decide WHERE the item is kept. To read, you must give this key exactly.

Sort key = order on the shelf

All items with the same partition key sit together, sorted by this value. It decides ORDER.

Primary key = the full address of an item. It has two forms:

FormMade ofFood app example
SimplePartition key onlyUsers table: PK = userId. One user, one item.
CompositePartition key + sort keyOrders table: PK = customerId, SK = orderTime. One customer, many orders, sorted by time.

Because same-PK items sit together and sorted, this query is very cheap:

// "Show my orders from this week, newest first"
Query:
  PK = "cust#42" AND SK > "2026-06-30"
  ScanIndexForward: false
Real system · WhatsApp-style chat (the classic pattern)

Every big chat system stores messages the same way: PK = chatId, SK = messageTimestamp.

Why does this work? When you open a chat, the app asks: "give me the last 50 messages of chat#123". That is ONE query, ONE partition, data already sorted by time. This is why chats open instantly even with billions of messages in the system. Discord and Facebook Messenger use this exact key pattern (on Cassandra/similar stores — same idea).

Real system · Lyft ride tracking

Lyft stores GPS points of every active ride in DynamoDB: PK = rideId, SK = timestamp. Millions of location writes per minute. Each write goes to exactly one partition (that ride's shelf). Reading a ride's route = one sorted Query. Simple keys, huge scale.

Section 3 · Storage

How data is stored inside

DynamoDB cuts your table into pieces called partitions. A partition is the real physical unit. Rules for each partition:

Hard limits per partition

Max ~10 GB data

Max 3,000 reads/sec (RCU)

Max 1,000 writes/sec (WCU)

Inside one partition

Data sits on SSD in a B-tree sorted by (PK, SK). Every write first goes to a write-ahead log (WAL) so nothing is lost in a crash.

Big table? No problem. A 100 TB table is just ~10,000 small partitions. Your query only ever touches one of them.

Try it: type a key, see which partition it goes to

Real system · Snapchat

Snap moved Snapchat's core storage (including Stories) to DynamoDB. Hundreds of millions of users, each posting and viewing snaps. Each user's data hashes to some partition. Because users are spread evenly across millions of partitions, no single machine ever holds "all of Snapchat" — each machine holds a small, easy slice. Snap also cut median latency significantly after the move.

You know Cassandra: same partition idea. Different disk structure — Cassandra uses LSM tree (memtable → SSTables → compaction). DynamoDB uses B-tree + WAL. B-tree = no compaction pauses = flatter P99.
Section 4 · Internal architecture

What happens when you place an order

Follow one write: customer 42 places an order in our food app. PutItem is called.

1App sends HTTPS request to DynamoDB
2Request Router checks auth, hashes "cust#42", finds which partition owns it
3Sends the write to that partition's leader replica
4Leader writes to its log, copies to 2 followers in other data centers (AZs)
5When 2 of 3 have saved it → "OK" goes back. ~few ms total.
Real system · Amazon Prime Day, hop by hop

On Prime Day, when 89 million requests hit per second, this exact path runs 89 million times per second. Why does it not fall over?

1. Routers are stateless. They hold no data. AWS just adds more router machines. Any router can serve any request.

2. Each write touches only 3 machines (one leader + two followers), no matter how big the table is. There is no "central" database that everything funnels through.

3. Millions of partitions work in parallel. 89M req/s spread over millions of partitions = a boring, small load per partition. Massive scale is really just massive parallelism.

Section 5 · Scalability

How it scales almost without limit

The secret is simple: nothing is global. Every limit, every leader, every lock is per-partition. To grow, DynamoDB just makes more partitions.

Partition split

A partition gets too big (10 GB) or too hot (traffic)? DynamoDB cuts it into two, moves halves to new machines, updates the routing map. Online, automatic. Capacity for that key range doubles.

Adaptive capacity

If one partition suddenly gets more traffic, DynamoDB gives it extra throughput and, if the heat continues, splits it. You do nothing.

Real system · Duolingo

Duolingo stores every user's learning progress in DynamoDB — around 31 billion items, serving hundreds of millions of exercises per day. Key = userId. Users spread evenly over partitions. When Duolingo's user base grew 10x, partitions just kept splitting in the background. The engineering team never re-sharded anything by hand. Compare that to MySQL, where 10x growth means painful manual sharding projects.

The one danger: hot partition key

Hashing spreads keys evenly, not traffic. If one single key gets most of the traffic, its one partition takes all of it. And a partition has a hard ceiling: 1,000 writes/sec, 3,000 reads/sec. Traffic above that gets throttled.

Real system · Flash sale / IPL ticket booking (the classic failure)

Imagine our food app runs a "₹1 biryani" flash sale for ONE restaurant. Table: Inventory, PK = restaurantId. Now 500,000 people hit the SAME key at once. One partition. 1,000 writes/sec limit. Everyone else gets ThrottlingException. The app "goes down" even though DynamoDB is healthy.

The fix used by real ticketing and sale systems — write sharding: split the hot key into 10 fake keys: rest#77#0 ... rest#77#9. Each write picks one at random. Now traffic spreads over 10 partitions = 10,000 writes/sec. To read the total, read all 10 and add them up. Same trick is used for vote counting, like counters, and live match scoreboards.

Section 6 · Indexing

GSI and LSI — querying by other fields

Problem: our Orders table has PK = customerId. So "orders of customer 42" is easy. But the restaurant also needs a screen: "show all MY incoming orders". Query by restaurantId? Not possible — it is not the key.

Answer: a Global Secondary Index (GSI). Simple mental model: a GSI is an automatic copy of your table with a different key. You write once; DynamoDB copies to the index in the background.

LSI (Local)GSI (Global)
What changesOnly the sort keyWhole key (new PK, new SK)
Lives whereSame partition as base dataIts own partitions (like a shadow table)
Read freshnessCan be strongAlways eventual (small lag)
When to createOnly at table creation, max 5Anytime, max 20
Food app useCustomer's orders sorted by amount instead of timeOrders by restaurant, orders by status, riders by zone
Real system · Every marketplace uses this 3-view pattern

One order must be visible from three sides — this is exactly one base table + two GSIs:

Customer view ("my orders") → base table, PK = customerId.
Restaurant view ("incoming orders") → GSI-1, PK = restaurantId.
Rider/ops view ("all PLACED orders in zone 5") → GSI-2, PK = zone#status.

Amazon's marketplace does the same thing at giant scale: one order is queryable by buyer, by seller, and by fulfillment status. Write once, DynamoDB maintains all views.

Two GSI traps (interviewers love these): 1) A GSI has its own capacity — if the GSI is too small, it throttles writes to the base table too. 2) A low-cardinality GSI key like status (only 3–4 values) creates hot partitions inside the GSI. Better: combine fields, like zone#status.
Section 7 · Availability & latency

How it stays up and stays fast

StrongLeader only1 RCUWallet balance, order status right after payment

Why it never goes down (99.99% SLA)

3 copies, 3 buildings

Every partition has 3 replicas in 3 different data centers (AZs). A whole building can burn down — 2 copies keep serving.

Leader dies → new leader in seconds

Followers notice missing heartbeats and elect a new leader (Paxos). Reads never stopped; writes pause only seconds.

Self-healing

A background service finds dead machines and rebuilds their partition copies on healthy machines. No human involved.

Global Tables

Optional: copy the whole table to other AWS regions, active-active. Survives a full region outage.

Why it stays fast at any size — the whole trick in one line

Every request does a small, fixed amount of work, no matter how big the table is.

O(1)routing map lookup (cached)
1 hopgo to that one machine

doneno JOINs, no query planner, no compaction pause
Real system · Disney+ watchlist & "continue watching"

Disney+ launched and gained over 10 million subscribers on day one. Your watchlist and your "resume from 42:10" bookmark are stored in DynamoDB, key = userId. Every play/pause writes a bookmark — a huge global write stream. Each write is one small item to one partition. Latency stays at a few milliseconds whether there are 10 million users or 150 million, because your request never touches anyone else's data.

Project setup & connection

📖 Launch week · The bill and the blowup
🔴 The incident

ShopSwift's first deploy to production fails instantly: ResourceNotFoundException: Table 'Orders' not found. The table clearly exists — the team can see it in the AWS console. Two days of confusion later, someone spots it: the app is configured for region us-east-1 (copied from a tutorial), but the table was created in ap-south-1 (Mumbai). Same week, a second surprise: the AWS bill shows charges nobody expected — every developer's laptop had been running tests against the real production AWS account for a month, because there was no separate local setup.

Why this happens: DynamoDB isn't a server you "connect" to like MySQL — every operation is a signed HTTPS call to a specific AWS region, and each region is a completely separate world. A table in Mumbai simply does not exist in Virginia. And tests hit real AWS by default because, unlike a database you install locally, DynamoDB lives in the cloud — unless you deliberately point development at a local stand-in, every test costs real money and touches real data.
💡 How Spring Boot + AWS SDK solve it, step by step
  1. Add one dependency (dynamodb-enhanced) — it brings both the low-level client and the Enhanced Client (typed Java objects).
  2. Set the region once, in one config class — every service in the app then talks to the right region automatically. No more copy-pasted tutorial regions scattered around.
  3. Add an endpoint override property — empty in production (real AWS), set to http://localhost:8000 in development (DynamoDB Local, a free container). Same code everywhere; only one property differs.
  4. Let credentials come from the environment — the SDK automatically picks up the server's IAM role in production and your AWS profile locally. No access keys ever appear in code or Git.
// pom.xml — Spring Boot 4, Java 25, AWS SDK v2
<dependency>
    <groupId>software.amazon.awssdk</groupId>
    <artifactId>dynamodb-enhanced</artifactId>
</dependency>
// DynamoDbConfig.java — one bean each, shared across the whole app
@Configuration
public class DynamoDbConfig {

    @Value("${aws.dynamodb.endpoint:}")   // empty in prod → real AWS; set locally → DynamoDB Local
    private String endpointOverride;

    @Bean
    public DynamoDbClient dynamoDbClient() {
        var builder = DynamoDbClient.builder().region(Region.AP_SOUTH_1);
        if (!endpointOverride.isBlank()) {
            builder.endpointOverride(URI.create(endpointOverride));  // e.g. http://localhost:8000 for dev
        }
        return builder.build();
    }

    @Bean
    public DynamoDbEnhancedClient enhancedClient(DynamoDbClient client) {
        return DynamoDbEnhancedClient.builder().dynamoDbClient(client).build();
    }
}

10.2

Table & key design → Entity classes + CRUD

📖 Week 3 · The ₹40,000 query
🔴 The incident

The "My Orders" page works fine for weeks. Then finance flags something odd: DynamoDB read costs jumped 15x in one month. The cause: "My Orders" was built with a Scan — because the Orders table's only key is orderId, and there's no direct way to ask "give me Meera's orders." So every page view reads the ENTIRE table — millions of items — and throws away everything that isn't Meera's. Slow for users, and DynamoDB bills for every item read, not every item returned.

Why this happens: in DynamoDB you can only ask fast, cheap questions along the key you chose when creating the table. Key on orderId alone and "find by customer" simply has no fast path — the database's only option is to read everything. The mistake wasn't the query; it was the key design three weeks earlier. In DynamoDB, you design the table around your questions, not around your data.
💡 How DynamoDB + Spring Boot solve it, step by step
  1. List the real questions first. The #1 question is "give me this customer's orders, newest first." So the key must start with the customer.
  2. Redesign the key as customerId (partition key) + orderTime (sort key). Now all of Meera's orders live together, already sorted by time.
  3. Map the item to a Java class with two annotations (@DynamoDbPartitionKey, @DynamoDbSortKey) and wrap the table in a small @Repository class.
  4. Replace the Scan with a Query. A Query touches only Meera's partition — pay for 12 items, not 12 million.

// Order.java — @DynamoDbBean = the Enhanced Client's version of @Document
@DynamoDbBean
public class Order {
    private String customerId;      // partition key
    private String orderTime;       // sort key — ISO timestamp, sorts naturally
    private String orderId;
    private String productId;
    private int quantity;
    private BigDecimal unitPrice;
    private String status;

    @DynamoDbPartitionKey
    public String getCustomerId() { return customerId; }

    @DynamoDbSortKey
    public String getOrderTime() { return orderTime; }

    // ...getters/setters for the rest
}
// OrderRepository.java — a thin wrapper class; DynamoDB has no official Spring "MongoRepository" equivalent
@Repository
public class OrderRepository {
    private final DynamoDbTable<Order> table;

    public OrderRepository(DynamoDbEnhancedClient enhancedClient) {
        this.table = enhancedClient.table("ShopSwift_Orders", TableSchema.fromBean(Order.class));
    }

    public Order save(Order order) {
        table.putItem(order);            // INSERT / overwrite
        return order;
    }

    public List<Order> findByCustomer(String customerId) {
        QueryConditional query = QueryConditional.keyEqualTo(
            Key.builder().partitionValue(customerId).build());
        return table.query(r -> r.queryConditional(query).scanIndexForward(false))  // newest first
                     .items().stream().toList();
    }
}

10.3

Capacity modes & bulk writes

📖 Month 2 · Big Billion Days
🔴 The incident

ShopSwift's first mega-sale. Two failures, same night. First: at midnight, traffic jumps 25x — and orders start failing with ProvisionedThroughputExceededException. The table was set up with a fixed capacity that made sense for normal days; the sale blew straight through it. Second: the night before, the catalog team updated prices on 40,000 products with a simple loop — one putItem call per product. It ran for over an hour and a chunk of the calls timed out, leaving some products at the OLD price during the sale.

Why this happens: (1) In provisioned mode, you promise AWS a traffic level, and AWS holds you to it — traffic above the promise gets rejected, by design. Nobody re-did the math for a 25x night. (2) The price loop was slow because each putItem is one full HTTPS round trip (~10–20ms). 40,000 sequential round trips = over an hour of pure network waiting. The work wasn't slow — the trips were.
💡 How DynamoDB + Spring Boot solve it, step by step
  1. Switch spiky tables to on-demand mode (PAY_PER_REQUEST) — DynamoDB then absorbs whatever traffic arrives and bills per request. No promise to blow through.
  2. Batch the bulk update with BatchWriteItem — up to 25 items per network trip. 40,000 items become 1,600 trips instead of 40,000.
  3. Handle partial failures — a batch can succeed for 23 items and return 2 as unprocessed. Production code must retry those, or prices silently stay stale (exactly what bit ShopSwift).
📄 The numbers that forced this decision
One-by-one loop40,000 putItem calls × ~15ms each ≈ 70+ minutes, timeouts under load, stale prices during the sale.
Batched40,000 ÷ 25 = 1,600 BatchWriteItem calls ≈ a few minutes, with unprocessed items retried automatically.
// ProductRepository.java — bulk price update using BatchWriteItem, with retry for leftovers
@Repository
public class ProductRepository {
    private final DynamoDbEnhancedClient enhancedClient;
    private final DynamoDbTable<Product> table;

    public void bulkUpdatePrices(List<Product> products) {
        for (List<Product> chunk : Lists.partition(products, 25)) {   // hard limit: 25 per batch
            WriteBatch batch = WriteBatch.builder(Product.class)
                .mappedTableResource(table)
                .addPutItem(builder -> chunk.forEach(builder::item))
                .build();

            var result = enhancedClient.batchWriteItem(r -> r.addWriteBatch(batch));
            var leftovers = result.unprocessedPutItemsForTable(table);
            if (!leftovers.isEmpty()) {
                bulkUpdatePrices(leftovers);      // retry what DynamoDB couldn't take this round
            }
        }
    }
}
// infrastructure-as-code note: table capacity mode (Terraform / CloudFormation, not Java)
resource "aws_dynamodb_table" "orders" {
  billing_mode = "PAY_PER_REQUEST"   // on-demand — no capacity promise to blow through
}

🔍 Code walkthrough
  1. Lists.partition(products, 25) respects the hard batch limit.
  2. unprocessedPutItemsForTable(table) asks "which items did DynamoDB NOT take?" — and the recursive call retries exactly those, until none remain.
  3. Same lesson as everywhere in this guide: batching beats one-at-a-time, and partial success must be handled, not assumed away.
10.4

Read consistency → strongly vs eventually consistent reads

📖 Month 3 · The disappearing cart item
🔴 The incident

A pattern emerges in support tickets: "I added the earbuds to my cart, opened the cart, and it was EMPTY. I refreshed and it appeared." Nobody can reproduce it in the office. It happens to maybe 1 in 500 add-to-cart actions, always in the same shape: write, then an immediate read that misses the write, then a refresh that finds it.

Why this happens: DynamoDB keeps three copies of every item in three data centers (the theory tab's Section 4). A write is confirmed once two copies have it — the third catches up milliseconds later. A default read may be answered by any of the three copies. Add-to-cart → instantly open cart is a race: if the read lands on the copy that hasn't caught up yet, the cart looks empty. It's rare and timing-dependent — which is exactly why the office could never reproduce it, and production, with millions of attempts, hit it daily.
💡 How DynamoDB + Spring Boot solve it, step by step
  1. Sort reads into two buckets. Bucket A: browsing, search results, product pages — a few ms of staleness is invisible. Bucket B: the cart right after a change, order status right after checkout — must reflect the write that JUST happened.
  2. Give bucket B one flag: consistentRead(true) — the read is answered by the up-to-date leader copy, never a lagging one. Costs 2x the read units; worth it exactly here.
  3. Leave bucket A on the cheap default — most of the app's reads, at half the cost.
📄 Same table, two very different reads
Product listing page (bucket A)Eventually consistent. A few ms of staleness is invisible — half the cost.
Cart, right after add-to-cart (bucket B)Strongly consistent. Must see the write that just happened — the 1-in-500 bug lives here.
// CartService.java — fixes the exact bug from the story
@Service
public class CartService {
    private final DynamoDbTable<CartItem> cartTable;

    public List<CartItem> getCartAfterUpdate(String customerId) {
        QueryConditional query = QueryConditional.keyEqualTo(
            Key.builder().partitionValue(customerId).build());

        return cartTable.query(r -> r.queryConditional(query)
                                      .consistentRead(true))     // forces the STRONG read
                         .items().stream().toList();
    }

    public List<Product> browseProducts(String category) {
        // default (eventually consistent) — cheaper, fine for browsing
        return productIndex.query(r -> r.queryConditional(
            QueryConditional.keyEqualTo(Key.builder().partitionValue(category).build())))
            .items().stream().toList();
    }
}
⚙️ What the one flag actually decides
SettingWhat it controls behind the scenesIf you change it
consistentRead(true)The read is served by the leader copy — the one that just accepted the writeGuaranteed fresh, costs 2× read units, slightly higher latency. Use it for read-your-own-write moments only.
consistentRead(false) — defaultAny of the 3 copies may answer; a lagging one is possible for a few msHalf the cost, spreads load. Using it everywhere is how the disappearing-cart bug shipped; using strong reads everywhere doubles the read bill for no user-visible benefit.
🔍 Code walkthrough
  1. The decision is per call, not app-wide — exactly like Mongo's ReadPreference.primary() vs secondaryPreferred() in the MongoDB guide. Same idea, different database.
  2. After shipping: the "empty cart" tickets stopped completely. Total read cost rose about 4% — only the cart path pays the premium.
10.5

Scaling → fixing a hot partition key

📖 Month 4 · The ₹999 flash sale
🔴 The incident

One-hour flash sale: Wireless Earbuds X200 at ₹999. Half a million people show up for ONE product. Within two minutes, checkout starts failing with ThrottlingException — even though the table is on on-demand mode and the AWS console shows the table as a whole is nowhere near any limit. The team is baffled: how can a table with near-unlimited capacity throttle?

Why this happens: capacity is unlimited for the table, but not for a single key. The Inventory table's partition key is productId — so every stock decrement for the earbuds hashes to the SAME partition, and one partition has a hard physical ceiling (~1,000 writes/second) that no billing mode can raise. On-demand mode scales by spreading traffic across many partitions — it cannot help when all the traffic insists on one key. This is the hot-partition problem from the theory tab's Section 5, live.
💡 How DynamoDB + Spring Boot solve it, step by step
  1. Split the one hot key into N fake keys — prod#5501#0 … prod#5501#9 — each holding a slice of the stock. This is called write sharding.
  2. Each write picks a shard at random — traffic now spreads over 10 partitions ≈ 10,000 writes/second of headroom.
  3. Guard every decrement with a condition (stock ≥ qty) so the last unit can't be sold twice.
  4. Pay on the read side instead — a stock check reads all 10 shards and adds them up. Stock checks happen far less often than decrements during a sale, so this is the cheap side to pay on.
📄 Before vs after write sharding
BeforeAll decrements → key prod#5501 → 1 partition → hard ceiling ~1,000 writes/sec → throttling in minutes.
AfterDecrements spread across prod#5501#0..9 → 10 partitions → ~10,000 writes/sec headroom.
// InventoryService.java — the write-sharding fix
@Service
public class InventoryService {
    private static final int SHARDS = 10;
    private final DynamoDbTable<InventoryShard> table;

    public boolean decrementStock(String productId, int qty) {
        String shardKey = productId + "#" + ThreadLocalRandom.current().nextInt(SHARDS);  // spread writes

        return table.updateItem(r -> r.item(new InventoryShard(shardKey, -qty))
            .conditionExpression(Expression.builder()
                .expression("stock >= :qty")                 // prevents overselling
                .putExpressionValue(":qty", AttributeValue.fromN(String.valueOf(qty)))
                .build())) != null;
    }

    public int totalStock(String productId) {
        int total = 0;
        for (int i = 0; i < SHARDS; i++) {
            InventoryShard shard = table.getItem(Key.builder().partitionValue(productId + "#" + i).build());
            total += (shard != null) ? shard.getStock() : 0;
        }
        return total;    // read fan-out — 10 reads instead of 1, done rarely
    }
}
DynamoDbEnhancedClient
⚙️ What each choice actually decides
ChoiceWhat it controls behind the scenesIf you change it
SHARDS = 10How many partitions the hot key's traffic spreads across — the write ceiling multiplierRaise to 20 → ~20,000 writes/sec headroom, but every stock check now does 20 reads. Size it to the biggest sale you actually expect, not the biggest imaginable.
Random shard pickEven spread with zero coordination between app instancesRound-robin per instance also works; anything based on customer ID would create its own mini hot spots for heavy users.
conditionExpression stock ≥ qtyCheck-and-decrement as ONE atomic step on the item itselfCheck in Java first, decrement second, and two buyers of the last unit both pass the check — the oversell bug in war story 10.10.
🔍 Code walkthrough
  1. shardKey = productId + "#" + random(0..9) — one line, and the hot key becomes ten warm keys.
  2. totalStock pays the fan-out cost where it's cheap: the "in stock" badge is read occasionally; decrements were happening thousands of times a second.
  3. Same trick powers likes counters, vote counts, and live scoreboards everywhere — spread the hot write, fan-in the rare read.
10.6

GSI & LSI → querying by other fields

📖 Month 5 · Two screens the key can't answer
🔴 The incident

Two feature requests land the same week. The ops team needs a live dashboard: "every order currently PROCESSING, across ALL customers" — so they can spot stuck orders. And the storefront needs "all products in Electronics." Both teams hit the same wall: the Orders key is customerId + orderTime, and Products is keyed by productId. Neither table can answer these questions without a full Scan — and after the ₹40,000 lesson in 10.2, Scan is banned.

Why this happens: a DynamoDB table answers fast questions ONLY along its key. Every genuinely new question ("by status", "by category") needs its own key — and that is literally what a GSI is: DynamoDB maintaining a second, automatically-synced copy of the data, organized under a different key. Nothing is wrong with the original design; the app simply grew new questions.
💡 How DynamoDB + Spring Boot solve it, step by step
  1. One GSI per new question. Orders gets status-index (key: status + orderTime). Products gets category-index.
  2. Declare it with two annotations on the entity — DynamoDB builds and maintains the copy in the background; every write to the base table flows into the index automatically.
  3. Query through the index handle (table.index("status-index")) exactly like querying the table.
// Order.java — adding a GSI is just two more annotations
@DynamoDbBean
public class Order {
    // ...existing fields from 10.2...

    @DynamoDbSecondaryPartitionKey(indexNames = "status-index")
    public String getStatus() { return status; }

    @DynamoDbSecondarySortKey(indexNames = "status-index")
    public String getOrderTime() { return orderTime; }   // reused as the GSI's sort key too
}
// OrderRepository.java — querying the new GSI
public List<Order> findByStatus(String status) {
    DynamoDbIndex<Order> statusIndex = table.index("status-index");
    QueryConditional query = QueryConditional.keyEqualTo(
        Key.builder().partitionValue(status).build());

    return statusIndex.query(r -> r.queryConditional(query))
                       .stream().flatMap(page -> page.items().stream())
                       .toList();
}
DynamoDbEnhancedClient
⚙️ What each GSI decision actually decides
DecisionWhat it controls behind the scenesIf you choose differently
GSI key = statusA shadow copy of the table, partitioned by status — "all PROCESSING" becomes a one-partition queryCareful: status has only ~5 values, so ALL processing orders share one GSI partition — at very high volume that's a hot partition inside the index. The fix is a composite key like status#region.
GSI capacityThe index has its own throughput, separate from the tableUnder-provision the GSI and something surprising happens: writes to the BASE table start throttling too, because every base write must also update the index. War story 10.10.
Reads through a GSIAlways eventually consistent — the copy syncs in the backgroundThere is no consistentRead(true) for a GSI. If a screen needs read-your-own-write freshness, it must query the base table.
🔍 Code walkthrough
  1. indexNames = "status-index" ties the annotation to a named index — one entity class can carry several GSIs.
  2. table.index(...) returns a query handle that works exactly like the table's — the ops dashboard code is four lines.
  3. The same order is now findable two ways — by customer (base key) and by status (GSI) — write once, DynamoDB maintains both views.
10.7

High availability & Global Tables

📖 Month 6 · The due-diligence question
🔴 The incident

ShopSwift is raising money. During technical due diligence, an investor's advisor asks one question: "AWS Mumbai had a major incident last year. If the whole region goes down on your biggest sale day, what happens?" The honest answer today: the entire business goes offline until AWS recovers. Every table, every copy, lives in one region.

Why this happens: DynamoDB's built-in 3 copies live in 3 different buildings (availability zones) — that survives fires and power failures, and it's why ShopSwift has had zero database downtime so far. But all 3 buildings are in the same city (region). A region-wide event — rare, but real — takes all three at once. Building-level safety and region-level safety are different problems.
💡 How DynamoDB solves it, step by step
  1. Enable Global Tables — one command adds a full, live replica of the table in another region (say Singapore). Writes flow both ways, typically syncing within about a second.
  2. Change zero application code — the same Enhanced Client code keeps working; the app doesn't know it's multi-region.
  3. Plan the one real consequence: with two regions accepting writes, a simultaneous write to the same item in both resolves as "last writer wins." The conditional writes from 10.5 and idempotent retries from 10.3 matter even more here.
// Enabling Global Tables — an infrastructure decision, not application code
aws dynamodb update-table \
  --table-name ShopSwift_Orders \
  --replica-updates '[{"Create": {"RegionName": "ap-southeast-1"}}]'   // add a Singapore replica
⚙️ What this actually decides
ChoiceWhat it controls behind the scenesIf you choose differently
Adding a replica regionA complete, writable copy of the table in another city, synced continuouslyCosts roughly double (storage + replicated writes in each region). Worth it once "the region is down" is a business-ending scenario — not before. ShopSwift enabled it only when investors made region failure a real question.
Doing nothing (single region)3 copies across 3 buildings in one city — 99.99% availabilityPerfectly fine for most businesses most of the time. The honest engineering answer is knowing WHICH failure you're paying to survive.
🔍 Walkthrough
  1. This is the rare chapter where the fix isn't Java — it's an infrastructure decision. The application's only job is to already be well-behaved: idempotent writes, conditional checks, retries.
  2. The due-diligence answer changed from "we go offline" to "traffic fails over to Singapore with about a second of replication lag."
10.8

Streams, DAX, TTL, Transactions

📖 Month 7 · Growing pains of success

Four real problems landed in the same sprint. Each has a one-paragraph story, a why, and a fix.

A) The warehouse finds out late → DynamoDB Streams

🔴 The incident

Warehouse packers work off a report refreshed every 5 minutes. On sale days, that's a 5-minute head start lost on every order — and "deliver tomorrow" promises start slipping.

Why this happens: refresh-based (polling) designs always trade delay for load — poll rarely and you're late, poll constantly and you hammer the table. The database KNOWS the instant an order lands; the warehouse just isn't being told.
💡 The fix

DynamoDB Streams emits an ordered event for every change; a Lambda receives each new order the moment it's written and pushes it to the warehouse screen. No polling anywhere.

// OrderStreamHandler.java — an AWS Lambda triggered by the Orders table's stream
public class OrderStreamHandler implements RequestHandler<DynamodbEvent, Void> {
    @Override
    public Void handleRequest(DynamodbEvent event, Context context) {
        for (DynamodbEvent.DynamodbStreamRecord record : event.getRecords()) {
            if ("INSERT".equals(record.getEventName())) {
                notifyWarehouse(record.getDynamodb().getNewImage());   // same role as Mongo Change Streams
            }
        }
        return null;
    }
}

B) The homepage hammers the same 20 items → DAX

🔴 The incident

During sales, the homepage's 20 "featured products" are read tens of thousands of times a second — identical reads, identical answers, full price paid every time.

Why this happens: the same barely-changing data is being re-fetched from the table on every page view. This is the textbook case for a cache — remember the answer, serve it from memory.
💡 The fix

DAX — a managed cache that speaks DynamoDB's own API. Swap the endpoint, keep the code; hot reads drop to microseconds and the table stops paying for repetition.

// DaxConfig.java — swap the endpoint, not the code
@Bean
public DynamoDbClient daxClient() {
    return DynamoDbClient.builder()
        .endpointOverride(URI.create("dax://shopswift-cluster.abcdef.dax-clusters.aws.com"))
        .build();
}

C) Abandoned carts pile up forever → TTL

🔴 The incident

Storage costs creep up month over month. Investigation: millions of cart items from shoppers who never checked out — some over a year old — sitting in the table forever.

Why this happens: nothing ever deletes them. Writing a cleanup job is easy to postpone forever; meanwhile every abandoned cart costs storage money monthly.
💡 The fix

A TTL attribute: stamp each cart item with an expiry time; DynamoDB deletes it automatically after that, for free — no cleanup job, no cron, no code to maintain.

@DynamoDbBean
public class CartItem {
    // ...
    private long expiresAt;   // epoch seconds — set to now + 7 days when the item is added

    @DynamoDbAttribute("expiresAt")
    public long getExpiresAt() { return expiresAt; }
}
// Table-level: aws dynamodb update-time-to-live --attribute-name expiresAt --enabled

D) The ₹2,499 that vanished → Transactions

🔴 The incident

Meera's wallet shows a ₹2,499 deduction for the earbuds — but no order exists. The app crashed between "deduct wallet" and "create order." 9 similar cases that month, always during deploys.

Why this happens: two separate writes with a millisecond gap between them — and a crash in the gap leaves write 1 done and write 2 never happening. Each write alone is safe; the PAIR isn't. Identical bug shape to QuickBite's ₹680 story in the MongoDB guide — this failure mode exists in every payment system in every database.
💡 The fix

TransactWriteItems — both writes commit as one all-or-nothing unit, with the balance check baked into the deduction itself.

// PaymentService.java
public void placeOrderWithPayment(String customerId, Order order, BigDecimal amount) {
    enhancedClient.transactWriteItems(r -> r
        .addUpdateItem(walletTable, TransactUpdateItemEnhancedRequest.builder(Wallet.class)
            .item(new Wallet(customerId, amount.negate()))
            .conditionExpression(Expression.builder()
                .expression("balance >= :amt")
                .putExpressionValue(":amt", AttributeValue.fromN(amount.toString()))
                .build())
            .build())
        .addPutItem(orderTable, order));   // both succeed, or both roll back
}
The pattern, once more, in a second database: a conditional check baked into the write, plus an all-or-nothing multi-write guarantee — the same shape as Mongo's @Transactional wallet fix and SQL's UPDATE ... WHERE balance >= amount. Different databases, one idea — say exactly that in an interview.

Looking ahead: single-table design

Every table so far — Orders, Products, Inventory, Carts, Wallets — has been separate. At real scale, DynamoDB teams often go one step further and combine related entities into one physical table, so one query can fetch a customer's profile and their recent orders together. That's exactly what the capstone below builds.

10.10

War stories — mistakes ShopSwift actually made

📖 Looking back

Overselling the last unit

Before the conditional write in 10.5, two customers could both "buy" the last Wireless Earbuds in the same millisecond — stock went to -1. Fixed with conditionExpression("stock >= :qty").

Scan in an admin report

An internal report used table.scan() to count today's orders — cheap in dev, brutally expensive once the table hit tens of millions of items. Fixed with a GSI on orderDate.

GSI throttling the base table

The status-index GSI was under-provisioned; once it throttled, writes to the base Orders table started failing too — a surprising, indirect failure mode worth knowing before it happens to you.

Forgetting BatchWriteItem's partial failures

Early bulk price-update code assumed all 25 items in a batch always succeeded — a few silently failed under load and were never retried, until someone noticed stale prices weeks later.

10.11

Capstone — ShopSwift's single-table design, end to end

📖 Month 9 · Putting everything together, the DynamoDB way
🔴 The incident

The "My Account" screen — Meera's profile, her recent orders, each order's payment status, together on one page — is built the obvious way: three tables (Customers, Orders, Payments), three separate queries per page load. It works, but on slow mobile connections the page visibly loads in three stages, and occasionally one of the three calls fails, showing a half-empty account page.

Why this happens: three tables means three round trips that can't be combined — DynamoDB has no JOIN. Each trip adds latency and its own chance of failure. The data isn't wrong; it's just spread across three places when the screen always wants it together.
💡 The DynamoDB-native answer: single-table design, step by step
  1. Use one table with deliberately generic key names — PK and SK — instead of entity-specific ones.
  2. Give related entities the SAME partition key. Meera's profile AND all her orders share PK = CUSTOMER#901, distinguished by their SK prefix (PROFILE, ORDER#...).
  3. Now one Query fetches her whole world — profile and every order together, in one round trip, sorted by SK.
  4. Entities looked up a different way get a different PK pattern — payments are fetched by orderId, so they live under PK = ORDER#... in the same table.
📄 One table, three entity types, one customer's world in one Query
 ShopSwift_Main — items sharing PK = CUSTOMER#901
PK              SK                    // item type, revealed by the SK prefix
CUSTOMER#901    PROFILE               // { name: "Meera Rao", email: "...", tier: "GOLD" }
CUSTOMER#901    ORDER#2026-07-06T13:00 // { orderId, productId, qty, total, status }
CUSTOMER#901    ORDER#2026-07-01T09:12 // an earlier order
ORDER#77213     PAYMENT               // { transactionId, method, status } — keyed by orderId, not customerId

One Query where PK = "CUSTOMER#901" returns the profile AND every order, in one call, sorted by SK. The three-stage page load became one instant load.

Step 1 — One item class for many entity types

// AccountItem.java — a generic item that can BE a profile, an order, or a payment
@DynamoDbBean
public class AccountItem {
    private String pk;
    private String sk;
    private String entityType;     // "PROFILE" | "ORDER" | "PAYMENT" — tells the app how to read the rest
    private Map<String, String> attributes;   // the entity-specific fields

    @DynamoDbPartitionKey
    public String getPk() { return pk; }

    @DynamoDbSortKey
    public String getSk() { return sk; }

    // ...getters/setters
}

Step 2 — Repository: one class, three kinds of query

// AccountRepository.java
@Repository
public class AccountRepository {
    private final DynamoDbTable<AccountItem> table;

    // The whole point: profile + all orders, ONE query
    public List<AccountItem> getCustomerWorld(String customerId) {
        QueryConditional query = QueryConditional.keyEqualTo(
            Key.builder().partitionValue("CUSTOMER#" + customerId).build());
        return table.query(r -> r.queryConditional(query))
                     .items().stream().toList();
    }

    public void saveProfile(String customerId, CustomerProfile profile) {
        table.putItem(AccountItem.of("CUSTOMER#" + customerId, "PROFILE", profile));
    }

    public void saveOrder(String customerId, Order order) {
        String sk = "ORDER#" + order.getOrderTime();
        table.putItem(AccountItem.of("CUSTOMER#" + customerId, sk, order));
    }

    public AccountItem getPayment(String orderId) {
        return table.getItem(Key.builder()
            .partitionValue("ORDER#" + orderId).sortValue("PAYMENT").build());
    }
}
DynamoDbEnhancedClient
🔍 What's actually happening here
  1. getCustomerWorld is the entire payoff — it directly fixes the three-stage page load with a single Query, because profile and orders share the same PK.
  2. entityType tells the service layer how to interpret each returned item — the app branches on it to rebuild a real CustomerProfile or Order object.
  3. saveOrder uses SK = "ORDER#" + orderTime — orders naturally sort chronologically under the profile, reusing 10.2's sort-key idea.
  4. getPayment deliberately uses a different PK pattern (ORDER#...) — payments are looked up by orderId, so they get their own partition in the same physical table.

Step 3 — Controller (the actual REST API)

// AccountController.java
@RestController
@RequestMapping("/api/v1/account")
public class AccountController {
    private final AccountRepository accountRepository;
    private final PaymentService paymentService;         // from 10.8 — transactWriteItems
    private final InventoryService inventoryService;     // from 10.5 — write-sharded stock

    @GetMapping("/{customerId}")
    public AccountSummary getAccount(@PathVariable String customerId) {
        return AccountSummary.from(accountRepository.getCustomerWorld(customerId));   // "My Account" screen
    }

    @PostMapping("/{customerId}/orders")
    public ResponseEntity<Order> placeOrder(@PathVariable String customerId,
                                             @RequestBody @Valid OrderRequest request) {
        boolean reserved = inventoryService.decrementStock(request.productId(), request.quantity());
        if (!reserved) {
            return ResponseEntity.status(HttpStatus.CONFLICT).build();   // out of stock — 10.5's conditional write
        }
        Order order = request.toOrder(customerId);
        paymentService.placeOrderWithPayment(customerId, order, order.getTotal());  // 10.8 — transaction
        accountRepository.saveOrder(customerId, order);
        return ResponseEntity.status(HttpStatus.CREATED).body(order);
    }

    @GetMapping("/orders/{orderId}/payment")
    public AccountItem getPaymentStatus(@PathVariable String orderId) {
        return accountRepository.getPayment(orderId);
    }
}

Horizontal scale

Every service above is stateless — ShopSwift runs many identical app instances behind a load balancer; DynamoDB itself never needs the app to "know" how many partitions exist.

One query, one round trip
