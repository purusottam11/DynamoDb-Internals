Section 0 · Why DynamoDB exists

The story: Amazon's shopping cart broke

In 2004, Amazon ran its shopping cart on Oracle (a relational database). During the holiday shopping rush, the database could not keep up. The site went down. Amazon lost money every minute.

Amazon's engineers realized something important: the shopping cart does not need SQL. It only needs two operations:

Put

"Save this cart for user 42."

Get

"Give me the cart of user 42."

No JOINs. No complex queries. Just key → value. So they built a simple key-value store that could scale by adding more machines. That system became the Dynamo paper (2007). Later, AWS turned it into a managed service: DynamoDB (2012).

Real system · Amazon.com itself

Today Amazon runs its shopping cart, order history, and huge parts of the retail site on DynamoDB. In 2019 Amazon shut down its last internal Oracle databases.

Prime Day 2021 numbers (from AWS's own paper): DynamoDB served trillions of API calls and peaked at 89.2 million requests per second — with single-digit millisecond latency. No human touched a server. That is the scale this guide explains.

Section 1 · Basics

What exactly is DynamoDB?

DynamoDB is a database from AWS. Three words describe it:

NoSQL (key-value)

You find data by its key. Like a giant HashMap that lives across thousands of machines. No JOINs, no SQL.

Fully managed

You never see a server. You just call an API. AWS handles machines, disks, replication, and failures.

Fast at any size

A read takes a few milliseconds. Whether your table has 1 GB or 100 TB — same speed. This is the main promise.

Not for everything

Bad for ad-hoc reporting, complex JOINs, "search by anything". You must know your queries before you design the table.

Real system · Zoom during COVID

In early 2020, Zoom's daily meeting participants jumped from 10 million to 300 million in about three months. A 30x jump. No team can buy and set up servers that fast.

Zoom used DynamoDB for user and meeting metadata. Because DynamoDB adds capacity by itself, Zoom scaled 30x without redesigning the database. That is what "fully managed" buys you: the scaling problem becomes AWS's problem.

Section 2 · Keys

Partition key and sort key — in plain words

Think of a huge library with thousands of shelves.

Partition key = which shelf

DynamoDB hashes this value to decide WHERE the item is kept. To read, you must give this key exactly.

Sort key = order on the shelf

All items with the same partition key sit together, sorted by this value. It decides ORDER.

Primary key = the full address of an item. It has two forms:

FormMade ofFood app example
SimplePartition key onlyUsers table: PK = userId. One user, one item.
CompositePartition key + sort keyOrders table: PK = customerId, SK = orderTime. One customer, many orders, sorted by time.

Because same-PK items sit together and sorted, this query is very cheap:

// "Show my orders from this week, newest first"
Query:
  PK = "cust#42" AND SK > "2026-06-30"
  ScanIndexForward: false
Real system · WhatsApp-style chat (the classic pattern)

Every big chat system stores messages the same way: PK = chatId, SK = messageTimestamp.

Why does this work? When you open a chat, the app asks: "give me the last 50 messages of chat#123". That is ONE query, ONE partition, data already sorted by time. This is why chats open instantly even with billions of messages in the system. Discord and Facebook Messenger use this exact key pattern (on Cassandra/similar stores — same idea).

Real system · Lyft ride tracking

Lyft stores GPS points of every active ride in DynamoDB: PK = rideId, SK = timestamp. Millions of location writes per minute. Each write goes to exactly one partition (that ride's shelf). Reading a ride's route = one sorted Query. Simple keys, huge scale.

Section 3 · Storage

How data is stored inside

DynamoDB cuts your table into pieces called partitions. A partition is the real physical unit. Rules for each partition:

Hard limits per partition

Max ~10 GB data

Max 3,000 reads/sec (RCU)

Max 1,000 writes/sec (WCU)

Inside one partition

Data sits on SSD in a B-tree sorted by (PK, SK). Every write first goes to a write-ahead log (WAL) so nothing is lost in a crash.

Big table? No problem. A 100 TB table is just ~10,000 small partitions. Your query only ever touches one of them.

Try it: type a key, see which partition it goes to

Real system · Snapchat

Snap moved Snapchat's core storage (including Stories) to DynamoDB. Hundreds of millions of users, each posting and viewing snaps. Each user's data hashes to some partition. Because users are spread evenly across millions of partitions, no single machine ever holds "all of Snapchat" — each machine holds a small, easy slice. Snap also cut median latency significantly after the move.

You know Cassandra: same partition idea. Different disk structure — Cassandra uses LSM tree (memtable → SSTables → compaction). DynamoDB uses B-tree + WAL. B-tree = no compaction pauses = flatter P99.
Section 4 · Internal architecture

What happens when you place an order

Follow one write: customer 42 places an order in our food app. PutItem is called.

1App sends HTTPS request to DynamoDB
2Request Router checks auth, hashes "cust#42", finds which partition owns it
3Sends the write to that partition's leader replica
4Leader writes to its log, copies to 2 followers in other data centers (AZs)
5When 2 of 3 have saved it → "OK" goes back. ~few ms total.
Real system · Amazon Prime Day, hop by hop

On Prime Day, when 89 million requests hit per second, this exact path runs 89 million times per second. Why does it not fall over?

1. Routers are stateless. They hold no data. AWS just adds more router machines. Any router can serve any request.

2. Each write touches only 3 machines (one leader + two followers), no matter how big the table is. There is no "central" database that everything funnels through.

3. Millions of partitions work in parallel. 89M req/s spread over millions of partitions = a boring, small load per partition. Massive scale is really just massive parallelism.

Section 5 · Scalability

How it scales almost without limit

The secret is simple: nothing is global. Every limit, every leader, every lock is per-partition. To grow, DynamoDB just makes more partitions.

Partition split

A partition gets too big (10 GB) or too hot (traffic)? DynamoDB cuts it into two, moves halves to new machines, updates the routing map. Online, automatic. Capacity for that key range doubles.

Adaptive capacity

If one partition suddenly gets more traffic, DynamoDB gives it extra throughput and, if the heat continues, splits it. You do nothing.

Real system · Duolingo

Duolingo stores every user's learning progress in DynamoDB — around 31 billion items, serving hundreds of millions of exercises per day. Key = userId. Users spread evenly over partitions. When Duolingo's user base grew 10x, partitions just kept splitting in the background. The engineering team never re-sharded anything by hand. Compare that to MySQL, where 10x growth means painful manual sharding projects.

The one danger: hot partition key

Hashing spreads keys evenly, not traffic. If one single key gets most of the traffic, its one partition takes all of it. And a partition has a hard ceiling: 1,000 writes/sec, 3,000 reads/sec. Traffic above that gets throttled.

Real system · Flash sale / IPL ticket booking (the classic failure)

Imagine our food app runs a "₹1 biryani" flash sale for ONE restaurant. Table: Inventory, PK = restaurantId. Now 500,000 people hit the SAME key at once. One partition. 1,000 writes/sec limit. Everyone else gets ThrottlingException. The app "goes down" even though DynamoDB is healthy.

The fix used by real ticketing and sale systems — write sharding: split the hot key into 10 fake keys: rest#77#0 ... rest#77#9. Each write picks one at random. Now traffic spreads over 10 partitions = 10,000 writes/sec. To read the total, read all 10 and add them up. Same trick is used for vote counting, like counters, and live match scoreboards.

Section 6 · Indexing

GSI and LSI — querying by other fields

Problem: our Orders table has PK = customerId. So "orders of customer 42" is easy. But the restaurant also needs a screen: "show all MY incoming orders". Query by restaurantId? Not possible — it is not the key.

Answer: a Global Secondary Index (GSI). Simple mental model: a GSI is an automatic copy of your table with a different key. You write once; DynamoDB copies to the index in the background.

LSI (Local)GSI (Global)
What changesOnly the sort keyWhole key (new PK, new SK)
Lives whereSame partition as base dataIts own partitions (like a shadow table)
Read freshnessCan be strongAlways eventual (small lag)
When to createOnly at table creation, max 5Anytime, max 20
Food app useCustomer's orders sorted by amount instead of timeOrders by restaurant, orders by status, riders by zone
Real system · Every marketplace uses this 3-view pattern

One order must be visible from three sides — this is exactly one base table + two GSIs:

Customer view ("my orders") → base table, PK = customerId.
Restaurant view ("incoming orders") → GSI-1, PK = restaurantId.
Rider/ops view ("all PLACED orders in zone 5") → GSI-2, PK = zone#status.

Amazon's marketplace does the same thing at giant scale: one order is queryable by buyer, by seller, and by fulfillment status. Write once, DynamoDB maintains all views.

Two GSI traps (interviewers love these): 1) A GSI has its own capacity — if the GSI is too small, it throttles writes to the base table too. 2) A low-cardinality GSI key like status (only 3–4 values) creates hot partitions inside the GSI. Better: combine fields, like zone#status.
Section 7 · Availability & latency

How it stays up and stays fast

Read choices — pick per request

ModeWho answersCostFood app use
Eventual (default)Any of the 3 copies0.5 RCURestaurant menu, reviews — tiny lag is fine
StrongLeader only1 RCUWallet balance, order status right after payment
Transaction2-phase across leaders2 RCUDeduct wallet + create order together (all or nothing)

Why it never goes down (99.99% SLA)

3 copies, 3 buildings

Every partition has 3 replicas in 3 different data centers (AZs). A whole building can burn down — 2 copies keep serving.

Leader dies → new leader in seconds

Followers notice missing heartbeats and elect a new leader (Paxos). Reads never stopped; writes pause only seconds.

Self-healing

A background service finds dead machines and rebuilds their partition copies on healthy machines. No human involved.

Global Tables

Optional: copy the whole table to other AWS regions, active-active. Survives a full region outage.

Why it stays fast at any size — the whole trick in one line

Every request does a small, fixed amount of work, no matter how big the table is.

O(1)hash the key → partition
O(1)routing map lookup (cached)
1 hopgo to that one machine
O(log n)B-tree search inside a ≤10 GB slice
doneno JOINs, no query planner, no compaction pause
Real system · Disney+ watchlist & "continue watching"

Disney+ launched and gained over 10 million subscribers on day one. Your watchlist and your "resume from 42:10" bookmark are stored in DynamoDB, key = userId. Every play/pause writes a bookmark — a huge global write stream. Each write is one small item to one partition. Latency stays at a few milliseconds whether there are 10 million users or 150 million, because your request never touches anyone else's data.

Real system · How the food app's order flow uses 4 features at once

1. Transaction: deduct wallet + create the order item + decrement stock with a conditional write (stock > 0) — all or nothing.

2. Stream: the new order item appears in DynamoDB Streams. A Lambda reads it and notifies the restaurant. No dual-write problem, no Kafka needed for this hop.

3. GSI: the restaurant dashboard queries the restaurant-index for its incoming orders.

4. TTL: the rider's live GPS items carry a TTL and clean themselves up after the delivery.

This is a complete, real production pattern — and a great answer skeleton for a system design interview.

One-line summaries to say in an interview

What is it

"A managed key-value store where the key is hashed to pick a partition, and every partition is a 3-replica Paxos group across 3 AZs."

Why it scales

"Nothing is global. Every request touches one partition and 1–3 machines, so scale = add partitions. Amazon proved it at 89M req/s."

Why it's fast

"Fixed work per request: O(1) routing + one B-tree seek in a ≤10 GB slice. No JOINs, no compaction pauses."

Its weakness

"Hot partition keys — hash spreads keys, not traffic. Fix with write sharding, better keys, or DAX."
